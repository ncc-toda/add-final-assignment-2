# MVP実装計画: 天気予報TUIツール(`weather`)

- 対象: `docs/requirements.md` に定義されたスコープ全体(フェーズ分けなし、一気実装方針)
- 実装主体: AIエージェント
- 進め方: 本ドキュメントの各タスクを1セッション相当の単位として、上から順に(依存関係が許す範囲で)実装する
- 開発手法: **t_wada式TDD(Red → Green → Refactor)**。ロジック部分は必ずテストファーストで進める。TUI描画そのものは要件どおり手動確認とする
- 各タスク着手時: このドキュメントでは大枠のみを決めており、詳細設計は着手のたびに **grilling スキルで深掘りしてから実装**する
- 完了したタスクはチェックボックスにチェックを入れて進捗を可視化する
- 実装中に仕様や技術選定の変更が生じた場合は `docs/requirements.md` も合わせて更新する

## 前提: 確定した技術スタック

`docs/requirements.md` 4.1節を参照。要点のみ再掲。

| 領域 | 採用technology |
| --- | --- |
| 言語 | Rust |
| TUI | `ratatui` + `crossterm` |
| HTTP | `ureq`(同期・軽量、非同期ランタイムなし) |
| CLI引数 | `clap`(derive) |
| シリアライズ | `serde` / `serde_json` / `toml` |
| 設定・キャッシュ配置 | `directories` |
| 地名サジェスト | `strsim` |
| エラー型 | 自前 enum + `Display`実装(外部クレートなし) |
| テスト | `cargo test` + `mockito`(HTTPモック) |

- ツール名/バイナリ名: `weather`(仮称)
- MVP対応地点: 都道府県庁所在地レベル(約47件)
- エリアコードマッピングデータ: JMA公式`area.json`等から事前生成し静的データとして同梱

## タスク一覧

### [x] Task 1: プロジェクト基盤構築

- **目的**: 以降の全タスクの土台となるRustプロジェクトの雛形を整える。
- **主な作業内容**:
  - `cargo init --bin`(バイナリ名 `weather`)
  - 依存クレート追加: `ratatui`, `crossterm`, `ureq`, `clap`, `serde`, `serde_json`, `toml`, `directories`, `strsim`(dev-dependenciesに`mockito`)
  - モジュール構成の雛形作成(例: `config`, `location`, `jma_client`, `cache`, `animation`, `cli`, `tui`, `error` など。詳細な分割は各タスク着手時のgrillingで調整可)
  - 共通エラー型(`enum AppError` + `Display`実装)の雛形を用意
  - `.gitignore` の整備(`target/` 除外。`Cargo.lock`はバイナリクレートのためコミット対象とする)
  - README雛形(ビルド・実行方法を後続タスクで追記していく前提の骨組み)
- **依存タスク**: なし
- **完了条件(DoD)**:
  - `cargo build` が通る
  - `cargo test` が(中身が空でも)通る
  - リポジトリに最低限のモジュール雛形とエラー型が存在する

### [ ] Task 2: 設定・キャッシュ基盤

- **目的**: TOML設定ファイルの読み書きと、10分TTLの汎用キャッシュ機構を用意する。
- **主な作業内容**:
  - `directories`によるconfig/cacheディレクトリ解決
  - TOML設定ファイルの読み込み・存在しない場合のデフォルト生成・書き込み
    - 設定項目: デフォルト地点、レイアウトプリセット名、カラーテーマ名、アニメーション設定(速度・密度・ON/OFF)
  - TTL付き汎用キャッシュ機構(任意のシリアライズ可能なデータをキー付きで保存/読込、有効期限判定)
- **TDD対象**: 設定のパース/デフォルト値生成ロジック、キャッシュの有効期限判定ロジック(ファイルI/O自体は薄いラッパーにし、ロジックを分離してテスト可能にする)
- **依存タスク**: Task 1
- **完了条件(DoD)**:
  - 設定ファイルの読み書き(デフォルト生成含む)の単体テストが揃っている
  - キャッシュの保存/読込/期限切れ判定の単体テストが揃っている

### [ ] Task 3: 地名→JMAエリアコード マッピング

- **目的**: ユーザーが入力した地名からJMAのエリアコードを解決できるようにする。
- **主な作業内容**:
  - 都道府県庁所在地レベル(約47件)のマッピング静的データを作成・同梱(JMA `area.json` 由来。生成方法の詳細は着手時にgrillingで詰める)
  - 地名の完全一致検索
  - 見つからない場合の類似地名サジェスト(`strsim`によるLevenshtein距離等の近似判定)
- **TDD対象**: 完全一致/該当なし/曖昧一致サジェストの各ロジック
- **依存タスク**: Task 1
- **完了条件(DoD)**:
  - マッピングデータ参照・サジェスト機能の単体テストが揃っている(正常系・地名なし系・タイプミス想定系)

### [ ] Task 4: JMA APIクライアント層

- **目的**: JMA非公式APIから予報・警報データを取得し、パースしてアプリ内のデータモデルに変換する。
- **主な作業内容**:
  - 短期予報(今日・明日、時間帯別)取得・パース
  - 週間予報(7日分、日別)取得・パース
  - 気象警報・注意報取得・パース
  - Task 2のキャッシュ機構と連携(10分キャッシュ、ネットワークエラー時はキャッシュへフォールバック+警告)
  - JSON構造変化時にクラッシュせずエラー型を返すハンドリング
  - 非公式APIへの配慮として、連絡先・ツール名が分かる`User-Agent`ヘッダーを設定する(requirements.md 3.1参照)
  - このタスクではエリアコードは検証済みの文字列として受け取るだけでよく、Task 3の地名解決ロジック自体には依存しない(呼び出し元であるTask 5で結合する)
- **TDD対象**: `mockito`によるHTTPモックを用いた正常系パース、ネットワークエラー時のキャッシュフォールバック、JSON構造異常時のエラーハンドリング
- **依存タスク**: Task 2
- **完了条件(DoD)**:
  - 正常系/キャッシュフォールバック/JSON構造異常の各ケースの単体テストが揃っている

### [ ] Task 5: CLIコマンド層

- **目的**: `weather` / `weather <地名>` / `weather --set <地名>` の引数体系を実装し、表示対象地点を確定させる。
- **主な作業内容**:
  - `clap`(derive)によるコマンドライン引数定義
  - 地点解決ロジック: 引数の地名があればTask 3で検索、なければTask 2の設定からデフォルト地点を使用
  - `--set <地名>` 実行時は地名を検証し、設定ファイルに保存して終了
  - 地名が見つからない場合はサジェストを表示して非ゼロ終了(TUI起動前のエラー表示)
- **TDD対象**: 引数パースの各分岐、地点解決ロジック(指定あり/デフォルト/該当なし/サジェスト)、`--set`保存呼び出しの単体テスト
- **依存タスク**: Task 2, Task 3(地名解決ロジック), Task 4(地点解決後の予報取得呼び出し)
- **完了条件(DoD)**:
  - CLIの主要分岐(引数なし/地名指定/`--set`/地名不正)の単体テストが揃っている(実際のTUI起動部分は対象外)

### [ ] Task 6: TUI情報パネル

- **目的**: 予報データを表示する情報パネル(タブ切り替え・警報強調表示・キー操作)を実装する。
- **主な作業内容**:
  - `ratatui`+`crossterm`によるフルスクリーン描画の骨組み
  - 今日・明日タブ / 週間予報タブの切り替え(キー操作)
  - 気象警報・注意報の強調表示(発令中のみ、赤系配色)
  - `q`キーでの終了
  - Task 2のレイアウトプリセット/カラーテーマ設定を反映する骨格(プリセットの具体的なバリエーションは着手時にgrillingで詳細化)
- **TDD対象**: タブの状態遷移など、描画に依存しないロジック部分は可能な範囲で単体テスト化する
- **手動確認対象**: 実際の画面描画・配色・強調表示・キー操作
- **依存タスク**: Task 2, Task 4(予報データモデル)
- **完了条件(DoD)**:
  - 手動でTUIを起動し、タブ切り替え・警報表示・`q`終了が確認できる
  - 状態遷移など切り出し可能なロジックの単体テストがある

### [ ] Task 7: TUI背景アニメーション

- **目的**: 天気に応じたASCIIアニメーション(晴れ・曇り・雨・雪・雷)を背景として描画する。
- **主な作業内容**:
  - JMA天気コード→5パターン(晴れ/曇り/雨/雪/雷)へのマッピングテーブル
  - 各パターンのASCIIアニメーションフレーム実装
  - 降水量・降水確率に応じた密度(アニメーション量)変化ロジック
  - Task 2のアニメーション設定(速度・密度・ON/OFF)を反映
  - Task 6の情報パネルと組み合わせ、背景+オーバーレイの構成に統合
- **TDD対象**: 天気コード→アニメーションカテゴリのマッピング、密度計算ロジック
- **手動確認対象**: 実際のアニメーション描画、密度変化、ON/OFF切り替え
- **依存タスク**: Task 2, Task 4(天気コード・降水確率データ), Task 6(パネルとの統合)
- **完了条件(DoD)**:
  - マッピング/密度計算ロジックの単体テストが揃っている
  - 手動確認で5パターンのアニメーション・密度変化・ON/OFF切り替えが動作する

### [ ] Task 8: 統合仕上げ・結合確認

- **目的**: 全モジュールを結線し、MVPとして一通り動作する状態に仕上げる。
- **主な作業内容**:
  - `main.rs`での結線: CLI引数解析 → 設定読込 → 地点解決 → APIクライアント呼び出し → TUI起動(情報パネル+背景アニメーション)
  - エラーケースの手動確認: ネットワーク切断時のキャッシュフォールバック、地名不正時のサジェスト表示、JSON構造変化のシミュレーション
  - README整備(ビルド方法・実行方法・設定ファイルの説明)
  - 一通りの利用シナリオのE2E的な手動QA(実際にJMA APIを叩いての動作確認)
- **依存タスク**: Task 1〜7 すべて
- **完了条件(DoD)**:
  - 主要シナリオ(初回起動/地点指定/`--set`/警報発令時/ネットワークエラー時)を手動で確認し、チェックリストを消化済み
  - READMEを見れば初見の人がビルド・実行できる状態になっている

## 進行時の注意事項

- 各タスクの着手時には、**grillingスキルで詳細な実装計画(データ構造・関数分割・テストケースなど)を練ってからTDDで実装する**。本計画では意図的にそこまでの詳細を詰めていない。
- タスクの依存関係上、Task 2・3・4は互いに独立しており並行して着手可能(Task 4はエリアコード文字列を受け取るだけでよく、Task 3の地名解決ロジックには依存しない)。両者はTask 5で結合される。Task 6・7は互いに統合が必要なため、実装順や分割粒度は着手時に再検討してよい。
- プリセットレイアウト/カラーテーマの具体的なバリエーションや、エリアコードマッピングデータの生成手順の細部は `docs/requirements.md` 7章の未決事項に該当するため、対応タスク(Task 3, Task 6)着手時に確定させ、必要に応じて要件定義書を更新する。
