# 要件定義書: 天気予報TUIツール

- ステータス: ドラフト(議論ベース、実装前)
- ツール名/バイナリ名: `weather`(仮称。MVP開発中はこの名称を採用し、将来変更可能とする)

## 1. 背景・目的

- 天気予報をターミナル上で確認できるツールが欲しい。
- 既存ツールは日本語対応が薄い、カスタマイズ性が低い、設定や実行コマンドが複雑、という不満がある。
- **「1コマンドで実行できるシンプルさ」**と、**「見た目のインパクト(TUIのビジュアル)」**を両立したツールを作る。
- 用途: 実務で使う社内/自分用ツール(社外配布は前提にしないが、将来配布する可能性を見据えて設計する)。

## 2. 対象ユーザー

- 自分自身、および社内メンバー。
- ターミナルでの作業が多く、天気を確認するためだけにブラウザ/アプリを開きたくない人。

## 3. 機能要件

### 3.1 天気データ取得

- データソース: **気象庁(JMA)非公式JSON API**を利用する。
  - APIキー不要・無料。
  - 日本語の予報文をそのまま利用できる。
  - 非公式仕様のため、将来の仕様変更リスクをエラーハンドリングで吸収する(3.7参照)。
  - **非公式APIへの配慮**: サーバー負荷をかけないよう、リクエストには連絡先やツール名が分かる`User-Agent`を設定する。また4.3節のキャッシュ(10分TTL)により、短時間の連続実行・複数地点切り替え時でも不要なリクエストが連発しないようにする。
- 取得情報:
  - 今日・明日の予報(時間帯別、JMAの短期予報API)
  - 週間予報(7日間、日別、JMAの週間予報API)
  - 気象警報・注意報(発令中は目立つ強調表示。例: 画面上部に赤系の色で表示)

### 3.2 地点指定

- 地点は**地名の文字列**で指定する(例: `weather 東京`)。
- 引数なしで実行した場合は、事前に保存されたデフォルト地点を表示する。
- デフォルト地点は `--set <地名>` のようなフラグで変更・保存する。
- 地名からJMAのエリアコードへの変換テーブル(マッピング)を内部に持つ。
  - **MVPでの対応粒度: 都道府県庁所在地レベル(約47件)**とする(例: 東京、大阪、札幌、福岡など)。JMAの一次細分区域(150件超)への対応は将来検討とする。
  - マッピングデータは、JMA公式の`area.json`等を元に**事前生成した静的データ**としてリポジトリに同梱する(実行時の追加API呼び出しは行わない)。
- 地名が見つからない/曖昧な場合はエラーハンドリングで対応する(3.7参照)。

### 3.3 表示する情報の範囲

- 今日・明日の予報(時間帯別の天気・気温・降水確率)
- 週間予報(7日分、日別の天気・気温・降水確率)
- 上記2つはTUI内でタブ的に切り替え可能。
- 気象警報・注意報(発令中のみ、強調表示)

### 3.4 TUIビジュアル

- 方向性: **アニメーション重視**。雨・雪などの天候をASCIIアニメーションで表現し、「只者じゃない」インパクトを演出する。
- レイアウト: **フルスクリーンの背景アニメーション + 情報パネルをオーバーレイ表示**。
  - 情報パネル(気温・降水確率・週間予報など)は可読性確保のため、背景色を敷いた領域に表示する。
  - パネルの外側や隙間からアニメーションが見える構成とし、「動きのある背景」と「読みやすい情報」を両立する。
- アニメーションパターン:
  - 基本5種: 晴れ・曇り・雨・雪・雷
  - JMAの天気コードから上記5種へのマッピングテーブルを作成する。
  - 降水量・降水確率に応じて雨/雪の密度(アニメーションの量)を変化させる。

### 3.5 コマンド体系(CLI)

- サブコマンドは持たず、**引数とフラグのみ**で完結させる。
  - `weather` : デフォルト地点の天気を表示
  - `weather <地名>` : 指定地点の天気をワンショット表示
  - `weather --set <地名>` : デフォルト地点を変更・保存

### 3.6 TUI内の操作性

- キー操作でタブ切り替え(今日・明日 ⇔ 週間予報)。
- `q` キーで終了。
- 常駐・自動更新は行わない(実行するたびに取得するワンショット型)。

### 3.7 エラーハンドリング

主要ケースを丁寧にカバーする:

- ネットワークエラー: 直近のキャッシュがあればフォールバック表示 + 警告メッセージ。キャッシュがなければエラーメッセージ。
- 地名が見つからない/不正: 類似地名をサジェスト表示する。
- JMA APIのJSON構造が変わった場合: クラッシュさせず、エラーメッセージを表示して終了する。

### 3.8 カスタマイズ(設定ファイル)

- 設定ファイル形式: **TOML**。
- カスタマイズ可能な項目:
  - デフォルト地点
  - **レイアウト**: 複数のプリセットレイアウト(例: 「フルスクリーンアニメ型」「ダッシュボード型」など)から選択。
  - **カラーテーマ**: 複数のプリセットテーマ(例: dark/light/ビビッド等)から選択。
  - **アニメーション**: 速度・密度などのパラメータ調整、ON/OFF切り替え。アニメパターン自体(晴れ/曇り/雨/雪/雷)はJMA天気コードに基づき固定。
- 自由座標指定やフルカスタムカラーパレット(RGB自由指定)などの「完全自由」なカスタマイズは行わない(実装コストとのバランスを取り、プリセット方式に統一)。

## 4. 非機能要件

### 4.1 技術スタック

- 実装言語: **Rust**
- TUIライブラリ: `ratatui`(確定)
- TUIバックエンド(端末制御): `crossterm`
- HTTPクライアント: `ureq`(同期・軽量。常駐しないワンショット型CLIのためtokio等の非同期ランタイムは採用しない)
- CLI引数パーサー: `clap`(derive機能を使用)
- シリアライズ/デシリアライズ: `serde` + `serde_json`(JMA APIレスポンス) + `toml`(設定ファイル)
- 設定・キャッシュディレクトリ解決: `directories`(`ProjectDirs`でOSごとの適切なconfig/cacheパスを取得)
- 地名の曖昧検索(サジェスト): `strsim`(Levenshtein距離などによる類似度判定)
- エラーハンドリング: 外部クレートは使わず、**自前の enum + `Display`実装**でエラー型を定義する
- テスト: 標準の`cargo test`。HTTPモックには`mockito`を使用してJMA APIレスポンスのパース処理をテストする

### 4.2 対応環境

- OS: macOS・Linux
- ターミナル: True Color対応の主要ターミナル(iTerm2, Terminal.app, Alacritty など)
- Windows対応は将来検討(初期スコープ外)

### 4.3 パフォーマンス・キャッシュ

- 実行のたびにAPIから最新情報を取得する。
- 短時間キャッシュ(目安10分)を設け、連打・連続実行時のAPI負荷を抑える。
- キャッシュはネットワークエラー時のフォールバック表示にも利用する。

### 4.4 品質保証

- APIレスポンスのパース処理や地名マッチングなどの**ロジック部分は単体テストを書く**。
- TUIの描画部分は手動確認とする(自動テスト対象外)。
- CI(GitHub Actionsなど)は現時点では構築しない。
- Git運用: mainブランチに直接commit・pushしていく(ブランチ運用・PRレビューは行わない)。
- 実装はAIエージェントが行い、**t_wada式TDD(Red→Green→Refactor)**でロジック部分の実装を進める。具体的な実装計画は `docs/plans/mvp_plans.md` を参照。

## 5. 配布・ライセンス

- 現時点では配布不要・プライベート運用。
- ただし将来的な配布(Homebrew tap + GitHub Releasesバイナリを想定)を見据え、環境依存を減らす設計とする。
- ライセンスは未決定(現状はプライベートのまま)。将来公開する場合はMIT等を検討。
